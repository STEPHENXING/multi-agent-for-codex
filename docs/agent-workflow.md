# 多 Agent 工作流细则

这个文件放详细流程，避免 `AGENTS.md` 变成制度手册。

## 角色分工

| 角色 | 职责 | 边界 |
| --- | --- | --- |
| 主 Agent | 协调、配置、集成验证、跨模块决策 | 不直接修改 `frontend/` 和 `backend/` 业务代码 |
| Planner | 需求拆解、数据库设计、API 契约 | 只输出规划，不写业务代码 |
| Frontend | React 页面、状态管理、接口调用 | 只修改 `frontend/` |
| Backend | FastAPI 接口、业务逻辑、数据模型 | 只修改 `backend/` |
| Reviewer | 审查契约一致性、质量风险、自检结果 | 不直接修业务代码 |

## 推荐循环

```text
需求输入
  -> Planner 输出 API 契约和任务拆解
  -> Frontend / Backend 并行开发
  -> Frontend / Backend 按固定格式返回结果
  -> Reviewer 审查
  -> 有 P1/P2 问题则回派对应子 Agent
  -> Reviewer 复审
  -> 主 Agent 集成验证
  -> 通过后结束
```

## 问题分级

| 级别 | 含义 | 处理方式 |
| --- | --- | --- |
| P1 | 阻塞功能或存在明显错误 | 必须修复 |
| P2 | 有质量风险或边界问题 | 优先修复，最多循环 3 次 |
| P3 | 优化建议 | 记录即可，不阻塞当前交付 |

## 集成验证清单

主 Agent 在 Reviewer 通过后执行：

1. 前端能否安装依赖并构建。
2. 后端能否安装依赖并通过测试。
3. API 代理、CORS、路由是否正常。
4. 核心流程是否能跑通，例如页面加载、健康检查、示例接口调用。
5. 配置文件是否正确，例如 `vite.config.ts`、FastAPI CORS、环境变量。
6. 是否存在 `\uXXXX` 中文转义残留。

## 回派规则

1. 前端构建、页面、交互、接口调用问题，回派 Frontend。
2. 后端构建、接口、测试、业务逻辑问题，回派 Backend。
3. API 契约不清或前后端理解不一致，回派 Planner。
4. 审查标准不清或误判，由主 Agent 决策。

## 人工介入条件

1. 同一个问题连续 3 次没有解决。
2. 问题跨越前端、后端、数据库，需要重新设计。
3. 环境、依赖、账号、密钥等外部条件缺失。
4. 子 Agent 修改范围越界。

## Git 建议

1. 一个原子任务一个 commit。
2. 提交信息使用 `<type>(<scope>): <说明>`。
3. 集成验证失败时优先使用 `git revert`，不要用 `git reset --hard` 抹历史。
4. 通过集成验证的节点可以打 tag，例如 `release/YYYY-MM-DD-01`。
